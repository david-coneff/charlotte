## AD-066: Light/dark theme toggle on the report + fix tracker
**Date:** 2026-06-27
**Problem:** the crawl report and the standalone fix tracker were dark-only; the operator wants a light option.
**Decision:** the palette is already CSS custom properties (`:root` = dark default), so a light theme is just
an override block on `html[data-theme="light"]`. Crucially it hangs off a data-ATTRIBUTE, NOT a class, so it
never collides with the no-flash tab restorer that owns `html.className` (`tab-<name>`). A new `--accent-fg`
variable carries the on-accent text colour (`#06121f` dark / `#ffffff` light) and replaces the 4 hard-coded
`#06121f` (active tab + accent buttons) that would otherwise be unreadable on the light accent. A fixed
top-right `.themebtn` (üåô/‚òÄÔ∏è) toggles `data-theme` + persists `charlotteTheme` in localStorage; a tiny head
script applies the saved choice before first paint (no flash). Shipped into BOTH report variants (single-site
+ multi-site index) via shared `THEME_LIGHT_CSS`/`THEME_HEAD`/`THEME_BTN`/`THEME_JS` consts, and inline into
the tracker ‚Äî constraint-clean (the toggle JS is single-quote + `+`-concat, no backtick/`${}`/backslash, so it
drops straight into TRACKER_TEMPLATE). The tracker's baked "Save copy" captures the live `data-theme` in its
`outerHTML`, so a shared copy opens in the same theme.
**Verification:** headless ‚Äî button present; a `dark‚Üílight‚Üídark` click round-trip flips `data-theme`, persists
`charlotteTheme`, and swaps the üåô/‚òÄÔ∏è icon; light-mode screenshots of the report and tracker show readable
contrast + white-on-accent tabs/buttons. Tracker template stays 0 backtick / 0 `${}` / 0 backslash; suite 170/0.

## AD-067: Two-level nesting in the fix tracker ‚Äî page/link sections roll up under a folder/domain parent
**Date:** 2026-06-27
**Problem:** the tracker grouped only at the finest level ‚Äî one section per referrer page (By page) or per
broken link (By broken link) ‚Äî with no coarser bucket, unlike the report (External by host;
Internal/Broken¬∑internal by first-level folder). With thousands of sections it leaned entirely on the
50/page pager. The operator wanted the report's roll-up: nest the sections under a collapsible folder/domain.
**Decision:** wrap the sections under a collapsible `.parent` keyed like the report ‚Äî the broken link's HOST
(External + By broken link) or the first-level path FOLDER (everything else, incl. the By-page referrer
pages, which are internal). `render`/`renderByLink` now emit `{p, html}` per section; `orderByParent` sorts
so same-parent sections are contiguous (bigger parents first, then alpha). `fillPanel` still paginates by
SECTION (PER_PAGE=50, so the DOM stays bounded no matter how lopsided the folders are) and wraps each page's
runs of same-parent sections in a `.parent` with a caret + "(N sections)" total ‚Äî a parent straddling a page
boundary simply repeats its header on the next page. Regex-free `hostOf`/`folderOf` (string ops only ‚Äî the
template forbids backslashes) compute the keys. Parent carets toggle `.collapsed`; "Collapse all" now
collapses parents (bird's-eye of folders/domains), "Expand all" opens parents + sections. The (page,link)
Fixed/verdict/notes wiring is untouched ‚Äî sections are just nested deeper and every `querySelectorAll` is
descendant-based, so refreshGroup/bulkFix/setVerdict and the By-page‚ÜîBy-link sync still resolve.
**Verification:** headless ‚Äî Internal/By-page parents = referrer folders (`/services-programs/ (2)`,
`/blog/ (1)`); Internal/By-link parents = broken-link folders (`/a/ (2)`, `/b/ (1)`); External/By-link
parents = domains (`www.facebook.com (2)`, `twitter.com (1)`), all count-desc; a parent caret collapses with
its sections still nested. Screenshot shows a folder parent wrapping the page sections. Tracker template stays
0 backtick / 0 `${}` / 0 backslash; full suite 170/0 (revtest's By-page‚ÜîBy-link sync passes through the nesting).

## AD-068: Inverted Fixed/Broken stat matrix on the fix tracker
**Date:** 2026-06-27
**Problem:** the tracker had no top-level scoreboard; the operator wanted the report's stat matrix, but keyed
to the tracker's TWO ORTHOGONAL axes ‚Äî BROKEN (does the link load = the verdict) and FIXED (has the page's
reference been remediated = the Fixed checkbox). They are independent: a link can be verdict-Broken with its
reference Fixed, or verdict-Working (no fix needed) etc.
**Decision:** a 2√ó3 `.statcard` above the tabs. **Bottom row = BROKEN (verdict-driven):** a link counts while
its verdict ‚âÝ Working; Broken internal/external destinations = distinct such links (per tab); Broken hyperlink
instances = the sum of their referrer pages. **Top row = FIXED (remediation-driven),** counted ONLY among
broken links so it's always a share of Broken: Fixed hyperlink instances = (page‚Üílink) pairs whose Fixed box
is ticked; Fixed internal/external destinations = broken links ALL of whose references are fixed. Each Fixed
card shows count + "(% of broken)". `recompute()` iterates `DATA.{internal,external}` reading the LIVE verdict
(`initVerdict`) + Fixed (`initChecked`), and runs from `progress()` ‚Äî so it fires on every Fixed tick,
All:Fixed, Clear-ticks, AND (newly wired) Broken/Working change. Marking a link Working drops it from BOTH the
broken counts and any fixed it contributed; marking it Broken again restores both (Fixed flags persist).
Inverted vs the report (Fixed on top, Broken below) because in the tracker the fix progress is the headline.
**Verification:** headless transition probe ‚Äî init `b:4/2/1 f:0/0/0`; fix A's two refs ‚Üí `f:2/1/0 (50%)` (A
becomes a fixed destination at 2/2); mark A Working ‚Üí `b:2/1/1 f:0/0/0` (broken + fixed both drop correctly);
un-Working ‚Üí `b:4/2/1 f:2/1/0` restored. Screenshot shows the green-Fixed-over-red-Broken matrix. Tracker
template stays 0 backtick / 0 `${}` / 0 backslash; full suite 170/0.

## AD-069: Export/Save buttons use the File System Access "Save As" picker (download fallback)
**Date:** 2026-06-27
**Problem:** every Export/Save button ‚Äî report: Export verdicts, Save shareable copy, Export fix tracker,
(legacy) allowlist export; tracker: Export, Save copy ‚Äî used the `<a download>` trick, which drops the file
into the browser's default Downloads folder with no choice of location or name. The operator wants a folder
picker. (Import already opens a picker via `<input type=file>`, so it was already fine.)
**Decision:** a `saveBlob(blob, name, okMsg)` helper tries `window.showSaveFilePicker()` (a real "Save As"
dialog ‚Äî operator picks folder + filename; the `types` accept-map is derived from the file extension), writes
the blob to the returned handle, then toasts success. Feature-detected: where the API is missing OR throws
(non-Chromium, or a restricted context), it falls back to the old `<a download>` path; cancelling the picker
(AbortError) is silent. This is exactly the additive, download-as-universal-fallback enhancement AD-034
deferred to "if revisited" ‚Äî localStorage auto-save is untouched. The report has TWO export IIFEs (the
allowlist/tracker-export script and the share-toolbar script) in separate scopes, so `dl`+`saveBlob` are
duplicated into both ‚Äî same pattern as the already-duplicated `toast`. The suite caught the first cut calling
`saveBlob` across the IIFE boundary (`saveBlob is not defined` ‚Äî a REAL ReferenceError that would also fire in
the browser, not just a test artifact). The tracker is one IIFE; its `saveBlob` is constraint-clean
(no backtick/`${}`/backslash).
**Verification:** `showSaveFilePicker` is present on `file://` in Chromium (probed = `function`). Headless with
a stubbed picker: clicking "Export fix tracker" calls
`showSaveFilePicker({suggestedName:'charlotte-fix-tracker.html', types:[{accept:{'text/html':['.html']}}]})`
and writes the 43 KB blob to the handle. With the picker absent (the DOM-stub suites) it falls back to
download ‚Äî full suite 170/0 including exporttest (13) and tracker3 (18).

## AD-070: Export filenames carry an auto-appended timestamp in the suggested name
**Date:** 2026-06-27
**Problem:** with the "Save As" picker (AD-069) the operator now names every export, but the *suggested* name
was a fixed string (`charlotte-fix-tracker.html`, `charlotte-verdicts-<host>.json`, ‚Ä¶). Re-exporting through
a day's triage either silently overwrites the previous file or forces the operator to hand-type a version
suffix every time. They wanted the picker to pre-fill a timestamped name so each export is naturally its own
versioned file.
**Decision:** stamp the timestamp inside `saveBlob(blob, name, okMsg)` ‚Äî at the very top, before either the
picker or the download branch ‚Äî so it covers BOTH paths and EVERY caller (Export verdicts, Save shareable
copy, Export fix tracker, allowlist export; tracker Export + Save copy) with one edit per scope. Format is
`<name>_YYYY-MM-DD_HH-MM_SS.<ext>` (e.g. `charlotte-fix-tracker_2026-06-27_14-03_09.html`), matching the
literal shape the operator asked for (date dash-joined, an underscore to the time, hour-minute dash-joined, an
underscore to seconds). Built from `new Date()` with a 2-digit `tz()` zero-pad ‚Äî no `Intl`/locale dependence,
deterministic, filesystem-safe (no colons/slashes). The insertion point is `name.lastIndexOf('.')`, NOT the
first dot, so a dotted hostname keeps its real extension (`tracker-www.example.com.html` ‚Üí
`tracker-www.example.com_<ts>.html`); a name with no extension just gets `_<ts>` appended. The timestamp
itself contains no dots, so a second `lastIndexOf('.')` downstream (the picker's accept-map) still resolves
the true extension. Lives in all THREE `saveBlob` copies ‚Äî report.js script A + script B (separate IIFE
scopes, byte-identical) and the tracker's single IIFE (constraint-clean: no backtick/`${}`/backslash).
**Verification:** extracted the exact stamping logic from the live template and ran it ‚Äî `report.json` ‚Üí
`report_2026-06-27_14-03_09.json`, `tracker-www.example.com.html` ‚Üí
`tracker-www.example.com_2026-06-27_14-03_09.html`, `NOEXT` ‚Üí `NOEXT_2026-06-27_14-03_09`. Format matches
`YYYY-MM-DD_HH-MM_SS` exactly. Tracker template stays 0/0/0; full suite 170/0.

## AD-071: Fix-tracker stat-card percentages adopt the report's adaptive `fmtPct` convention
**Date:** 2026-06-27
**Problem:** the tracker's Fixed stat cards showed their "% of broken" via `Math.round(num/den*100)` ‚Äî a whole
percent. On a real site the fixed share early on is a tiny sliver of the broken total (e.g. 1 fixed reference
among 5,000 broken instances = 0.02%), and whole-percent rounding collapses every such value to a flat,
misleading `0%`, hiding real progress. The report's broken-stats already solved this (AD-056) with an adaptive
formatter; the tracker should match it for a single percentage convention across both surfaces.
**Decision:** drop in the report's exact `fmtPct` next to `recompute()`'s `setP` and route the percentage
through it: `function fmtPct(p){if(!(p>0))return '0.0';var d=1;while(d<10&&Number(p.toFixed(d))===0)d++;return
p.toFixed(d);}`, with `setP` now emitting `'('+fmtPct(num/den*100)+'%)'`. Behaviour: a floor of **one** decimal
always (`50.0%`, `100.0%`, and `0.0%` for a zero/empty share), expanding the precision one digit at a time
(capped at 10) only while the value would still render as `0.0` ‚Äî so a non-zero-but-tiny share keeps its first
significant digit (`0.02%`, `0.0001%`) instead of vanishing. Identical wording/logic to report.js's `fmtPct`,
so the two stay in lockstep. Constraint-clean (single-quoted `'0.0'`, no backtick/`${}`/backslash).
**Verification:** extracted the template's `fmtPct` and exercised the convention ‚Äî `0/4‚Üí0.0%`, `5/10‚Üí50.0%`,
`3/7‚Üí42.9%`, `1/800‚Üí0.1%`, `1/10000‚Üí0.01%`, `1/1000000‚Üí0.0001%`. End-to-end headless render with one fixed
reference seeded among 5,000 broken instances: the live `#st-fInstP` card reads `(0.02%)` where the old
`Math.round` would have shown `(0%)`, and `#st-fIntP` reads `(0.0%)` (one-decimal floor). Full suite 170/0.

## AD-072: Batch per-page mini-tracker export (scoped, sanitized filenames) for delegating fixes
**Date:** 2026-06-27
**Problem:** the fix tracker's Import already merges the FULL per-link state (fixed flags + fixed-on,
verdicts + last-tested, notes ‚Äî everything under `cwfix:<host>:`), keyed per `(referrer page -> broken
link)` pair, so several people's exports accumulate cleanly. What was missing was the *fan-out* side: a way
to hand each page-owner a tracker for **just their page** instead of the whole-site file. Goal: from the
central tracker, batch-produce one self-contained mini-tracker per referrer page, named after the page so
they can be distributed, fixed, and re-imported with no manual re-ticking.
**Decision:** a **üóÇ Per-page** toolbar action (`savePerPage`) that, for every referrer page with at least
one still-not-Working link, bakes a mini-tracker scoped to that page. Scoping reuses `groups()` (already
inverts entries to `ref -> [{broken,...}]`), rebuilding each entry as `{url,reason,refs:[P],v,ts}` so the
mini's DATA carries ONLY P's links. The output shell is a **clone of the live document with the rendered
group lists blanked** (`#panel-int/#panel-ext/#pager-*` emptied) so no other page's links ride along and the
files stay small ‚Äî `fill()` rebuilds the scoped view on open. The scoped DATA is spliced into the shell
between two `/*CW_DATA_BOUNDS*/` comment markers added around the `var DATA = "__DATA__"` injection point (so
a running tracker can re-scope itself; markers survive `report.js`'s `tpl.replace('"__DATA__"', ‚Ä¶)`). Each
mini also gets a `__CW_TRK_SEED__` island holding **only P's state keys** (`scopedSeed` filters
collectState by: `vd:`/`vt:` for the broken URLs P references, `ft:`+bare-pkey+`n:` for ref===P), so current
progress carries and re-fan-out is faithful. Batch delivery uses **`showDirectoryPicker()`** (operator picks
one folder, all files written via `getFileHandle(create) -> createWritable`); where the directory API is
absent it falls back to sequential `<a download>`s. Filenames come from `pageFileName(url)`: drop the scheme,
then map every char outside `[A-Za-z0-9-._]` to `_` (so slashes and `?:&=` become underscores, "looks
similar" per the request), collapse repeats, trim, cap 120, dedupe with `-2/-3`. Round-trip works because the
mini's host == central's host and its fix keys are the identical `ref+NL+broken` pairs, so Import (applyState,
a per-key merge) drops them straight in. Constraint-clean (no backtick/`${}`/backslash; backslash via `BS`,
the `<scr'+'ipt>` split, and `<`‚Üí`BS+u003c` seed-escaping all mirror `saveCopy`).
**Verification:** new `pagetest` (10) unit-checks `pageFileName` (slashes‚Üí`_`, scheme dropped, length cap,
no `/` in output) + the marker placement. New `pageE2E` (13) drives the real feature in headless Chromium via
synchronous-thenable `showDirectoryPicker`/`Blob` stubs: a 3-page fixture (A‚Üí{u1,u2}, B‚Üí{u1,e1},
C‚Üí{working-only}) yields exactly TWO files `x_pageA.html`/`x_pageB.html` (C skipped), each scoped DATA holds
only that page's links (A:int[u1,u2]/ext[]; B:int[u1]/ext[e1]), refs reduced to the single page, panels
blanked, host preserved. Opening the scoped pageA mini renders `bInst=2,bInt=2` (the count being 2 not 3 is
itself proof no other page's link leaked into DATA). Tracker template stays 0/0/0; full suite 193/0
(170 core + 10 pagetest + 13 pageE2E). Note: a first test cut whole-text-searched the minis for foreign URLs
and false-positived ‚Äî `savePerPage` clones the live doc, which at click time includes the probe `<script>`
whose own source contained those URL literals; the authoritative no-leak signal is the parsed scoped DATA and
the `bInst` count, not a substring scan of the cloned document.

## AD-073: Team consolidation path ‚Äî multi-file Import + namespace-guarded merge + SharePoint/Power-Automate guide
**Date:** 2026-06-27
**Problem:** with per-page minis (AD-072) fanned out, the *return* path was still "email one JSON, Import one
file." The user asked whether the tracker could instead live on internal SharePoint and consume JSON dropped
into a shared folder via REST. Reality check: on a locked-down tenant **custom script is disabled** (an
inline-`<script>` HTML can't execute from a doc library; the supported route, SPFx, needs a build pipeline and
breaks the single-file design) and **cross-origin SharePoint REST is CORS-blocked** (would need Azure AD +
MSAL). So a browser-side REST consumer isn't viable there. The robust shape is server-side: contributors drop
files, **one consumer merges**, the central tracker reads the result ‚Äî which also dodges the multi-writer race
of a single shared state file.
**Decision:** chose the **Power-Automate-merge** architecture (the tracker stays a viewer) and delivered three
things. (1) **Multi-file Import** in the tracker: the `cwImpF` input gains `multiple`; a new `importStateFiles`
reads N files, applies each valid same-host one (`app` + `host` checked per file), tallies applied/skipped, and
reloads **once**. This is the pure-`file://` no-flow fallback (multi-select a whole inbox folder). (2)
**Namespace hardening** of `applyState`: it now writes only keys starting with `NS` (`cwfix:<host>:`), so an
ingested/merged file ‚Äî trusted or not ‚Äî can never inject stray localStorage keys. (3) `merge-fix-state.js`
(repo root, dependency-free): the **reference implementation** of the union merge AND an unattended CLI
(`node merge-fix-state.js --out state.json inbox/*.json`) for when even Power Automate is restricted ‚Äî same
semantics as the flow (one host; first valid file sets it, different-host skipped; namespace-guarded; **later
file wins** on key collision, mirroring WDL `union(current.v, incoming.v)`). Plus `SHAREPOINT-MERGE.md`: the
end-to-end build guide (library layout `inbox/`/`archive/`/`state.json`; trigger with **parallelism = 1** to
serialize merges; the `@union(...)` merge step; first-run handled by pre-seeding an empty `state.json`;
optional flow-side key filter; conflict = last-arrival-wins with an Office-Script/Function escape hatch for
newest-by-timestamp; three ways to hand the consolidated state back, incl. a flow-baked `__CW_TRK_SEED__` HTML).
The whole loop needs **no new tracker code to consume the flow's output** ‚Äî the mini Export and the central
Import already speak the same `{app,host,v}` contract; the merge is just a dictionary union.
**Verification:** new `merge-test` (12) covers the reference merger ‚Äî union of disjoint fixes, **later-wins** on
a shared `vd:` key, host isolation (different-host keys excluded), NS guard (stray key dropped), report counts,
importState-shaped output, and the from-disk CLI path incl. an unreadable file counted as skipped. CLI smoke:
`node merge-fix-state.js --out` writes a clean consolidated file + stderr summary. `tracker3-test` gains a
**C2 multi-file** section (6): two same-host files merge together, the recipient's own mark is preserved, a
stray non-`cwfix` key is NOT written (NS guard), a wrong-host file in the same batch is skipped, and the whole
batch triggers exactly one reload. Tracker template stays 0/0/0. Full suite **211/0** (was 193 + 6 multi-import
+ 12 merge). Note: the locked-down-tenant constraints (custom script, CORS) are the real design driver here ‚Äî
documented so the "why not just fetch the folder from the page" question doesn't get reopened.

## AD-074: Tracker batch-export by page OR subfolder, distinct referrer-page counting, + a Pages workload column
**Date:** 2026-06-27
**Problem:** three related fixes after the per-page export (AD-072) shipped. (1) The tracker subtitle reported
`ci.pages + ce.pages` referrer pages ‚Äî the SUM of the internal-tab and external-tab page counts, which
double-counts any page that links both a broken internal AND a broken external destination; this made the
subtitle (e.g. 551) disagree with the per-page export's file count (e.g. 512, a deduped union minus
all-Working pages). (2) Operators wanted to delegate a whole site SECTION to one owner, not just a file per
page. (3) The stat matrix counted links (instances/destinations) but not *pages*, so it under-described the
real workload ‚Äî "how many distinct pages (and owners) still need attention."
**Decision:** (1) subtitle now counts **distinct** referrer pages via a `distinctRefPages()` union over both
lists (named function, NOT an inner IIFE ‚Äî see verification). (2) `savePerPage` generalized to
**`saveBatch(mode)`**: `mode==='page'` keeps one file per page; `mode==='folder'` groups pages by
`folderOf` (tier-1 path folder) so every page under e.g. `/blog/` lands in one file, scoped to all those
pages' broken links and named after the folder. A shared `collect(g,pageSet)` gathers the broken links any
page in the set references, reducing each entry's `refs` to that set; `scopedSeed` now takes a pageSet (not a
single page). Two descriptive buttons ‚Äî **üóÇ Bulk export: per page** / **üóÅ Bulk export: per subfolder** ‚Äî
wire to `saveBatch('page')`/`saveBatch('folder')`; toasts use a per-mode noun and report skipped (all-Working)
groups. (3) A fourth stat-matrix column **Pages with broken links** (bottom) / **Pages remediated** (top):
`recompute` now also folds a `pg[referrer]` map ‚Äî a page is "broken" once it has any non-Working link and
"remediated" only when EVERY such link is Fixed (across internal + external) ‚Äî emitting `st-bPg`/`st-fPg` with
the same adaptive `%`. Grid widened `repeat(3,1fr)` ‚Üí `repeat(4,minmax(0,1fr))`.
**Verification:** new headless `trk-pages` probe ‚Äî A links 2 broken (both fixed), B links 1 (unfixed), C links
a Working-only link ‚Üí `Pages with broken links=2`, `Pages remediated=1`, `(50.0%)` (C correctly excluded).
New `folderE2E` (10) drives `saveBatch('folder')`: pages under `about/` collapse into one `x_about.html` with
both pages' links (each ref'd by its own page), `blog/`‚Üí`x_blog.html`, root‚Üí`x.html`, an all-Working `legacy/`
skipped. `pageE2E` (per-page, now `saveBatch('page')`) still green. tracker3 gains a subtitle case (a
cross-linking page counts as 1, not 2). **Gotcha:** the first cut wrote `distinctRefPages` as an inner IIFE
`(function(){‚Ä¶})()`; its `})();` is the first such token after the main IIFE's `(function(){`, so the tests'
slice-extraction (`indexOf("})();")`) cut the IIFE short ‚Üí "Unexpected end of input" in revtest + tracker3
(module-load still passed, since the template is just a string). Fixed by using a named function. Lesson:
**no inner IIFEs in the template** ‚Äî the slice-based test harness treats the first `})();` as the IIFE end.
Tracker template stays 0/0/0; full suite 222/0.

## AD-075: Report ‚Äî "Referrer pages with broken links" card, relocated outline legend, global link reuse, "triaged" wording, fuller broken-destination labels
**Date:** 2026-06-27
**Problem:** four report-side refinements. (1) The stat matrix counted broken instances/destinations but
not the distinct *pages* that carry broken links ‚Äî the "how many pages / owners must I chase" view; an
operator also asked for it explicitly. (2) The grey "Outline key" legend ate a stat-matrix slot. (3) An
operator reported the satellite reused-window behaviour "not applied to the External tab"; audit showed all
anchors already route through one global `target=_blank` interceptor (25/25 anchors), but the invariant was
implicit. (4) "Broken ¬∑ internal/external" cards risked being read as instance counts, and "tested" was
judged a less-descriptive term than "triaged" for the verdict action.
**Decision:** (1) replaced the legend card with a **Referrer pages with broken links** stat (`brokenPgN`):
server-renders an initial distinct-referrer-page count (`brokenRefPagesInit`, a Set over `activeInt`+`activeExt`
refs), then `recomputeBroken` keeps it live by accumulating into a `pset` the referrer pages (from a memoized
`refMap()` over `window.__CW_BROKEN__`) of every row that still counts as broken; the card carries the same
green/amber **triage-completeness** outline (`setTestState(brokenPgN, iT+eT, iN+eN)`) and `bad` class as the
other broken stats, no `%`. (2) the outline legend moved to a compact **fixed `.leghint` strip in the
upper-right by the theme toggle** (`LEGEND_HINT`, shown only when `hasTriage`). (3) NEWWIN's click handler now
intercepts **any `http(s)` anchor** (not just `target=_blank`) so every tab's links reuse the one satellite
window as an invariant ‚Äî and lets **modified clicks** (ctrl/cmd/shift/middle) fall through to a real new tab;
`blob:`/`#`/relative `file://` anchors are left alone. (4) cards relabelled **Broken internal/external
destinations**; a project-wide **tested‚Üítriaged** wording sweep (legend, stat tooltips, the "Manually
triaged X / N" counter, the "Last triaged" column header, the domain "all triaged"/"triaged K/N" indicators)
‚Äî CSS class names (`tested-all`/`tested-partial`/`untested`/`domalltested`) deliberately left unchanged
(internal, wired in JS+CSS).
**Verification:** new headless probes ‚Äî `pgcard-probe` (load: live recompute via REFMAP keeps `brokenPgN=4`
matching the server init, `__CW_BROKEN__` present, amber `tested-partial` + `bad` before triage) and
`pgcard-dyn` (after marking every link Working: `brokenPgN‚Üí0`, outline `tested-all` green, no `bad`). Static

## AD-083: Render-to-discover mode (`crawl-render.js --discover`) + `crawl.js --seeds`
**Date:** 2026-06-28
**Problem:** `crawl.js` is a static crawler ‚Äî it reads the bytes the server sends and never runs
JavaScript (`src/parse.js` regexes `<a href>` out of the raw response body; `src/fetch.js` returns it
unexecuted). On a JavaScript-built site (an SPA ‚Äî Laserfiche WebLink, SharePoint, most modern document
portals) the folder/navigation links aren't in that HTML; they're injected after an XHR. So the crawl
stalls after the few static chrome links and never reaches the documents. Reported against the Idaho
DHW WebLink repository, where the crawl stopped at ~2 links and never entered the visible folders.
`crawl-render.js` already owned a real headless browser (Playwright) but only as a *verifier* of links
already found ‚Äî it could not *discover* the links a render would reveal.
**Decision:** add a second mode to `crawl-render.js` (kept there so `crawl.js` stays zero-dependency at
runtime; Playwright remains an optionalDependency):
- **`--discover`** ‚Äî a rendered BFS. Each page is opened in real Chromium, the JS is allowed to settle
  (`--wait-until` defaults to `networkidle`, plus optional `--settle`), and links are harvested from the
  **live DOM** (`a[href]`, already absolute) with a best-effort `onclick`/`data-*` URL scan for entries
  that navigate via JS handlers (`--no-onclick-scan` to disable). Links are classified ‚Äî assets ignored,
  documents (pdf/office/‚Ä¶) recorded as leaves, in-scope HTML pages recursed ‚Äî honoring `--scope`
  domain|path, `--path-prefix`, `--include-subdomains`, `--max-depth`, `--max-pages`. Fragments that
  carry SPA routing state (`#?id=‚Ä¶`, `#/‚Ä¶`) are kept distinct (where the static crawler collapses all
  fragments). Output: a flat **`--seeds`** file (in-scope pages + documents) plus a JSON manifest
  (`--out`, default `crawl-render.discover.json`) and an optional `--html` summary. `--http-fallback`
  degrades to a *static* harvest that honestly reports it can't see a true SPA's JS-built links.
- **`crawl.js --seeds FILE`** (in `src/cli.js`) ‚Äî newline-delimited start URLs (`#` comments ignored).
  Marked `seedMode`, which makes `crawl()` seed the frontier from **every** URL at depth 0 in **one**
  combined crawl (shared seen-set, single report) rather than the per-URL multi-site index ‚Äî the natural
  meaning of a seed list, and it avoids an N-report explosion when discovery returns hundreds of
  same-host URLs. The hand-off is `node crawl.js --seeds FILE --max-depth 0 --check-external` (verify +
  document-scan the discovered set; discovery already did the traversal). `crawl.js` is the generated
  roll-up of `src/`, rebuilt via `npm run build`.
**Verification:** 16 pure-helper unit tests (canonicalize/scope/document/asset/onclick-scan/static-href)
all pass. End-to-end against a local SPA fixture that injects its folder links *after* load (the WebLink
failure mode): static `crawl.js` reaches 0 folders / 0 documents (reproduces the report), while
`--discover` traverses the full tree (depth 0‚Üí1‚Üí2) and finds all 4 documents, writing 9 seeds; feeding
those to `crawl.js --seeds --max-depth 0` yields one report that fetches all 9 and scans all 3 PDFs
(extracting their internal `/URI` links). Regressions checked: plain multi-URL still produces a
multi-site index; `--http-fallback --discover` runs and correctly finds no SPA links. Lesson (re)applied:
no backtick literals inside the help **template literal** ‚Äî an inner `` `‚Ä¶` `` broke the `src/cli.js`
build until quoted (cf. AD-074). **Follow-up:** the `CRAWLER/` rhiz-Merkle reference still describes
`crawl-render.js` as verify-only; its discover section is left for the next mechanical `rhiz docs`
partition pass rather than hand-editing the integrity-hashed parts. README + both `--help` texts are
current.
**Addendum (2026-06-28):** added `crawl-render.js --discover --ignore-case` (off by default) ‚Äî folds path
capitalization and sorts query params in the dedup *key only* (the original-cased URL is still
fetched/recorded), so an IIS/ASP.NET site's `/Browse.aspx?b=2&a=1` and `/browse.aspx?a=1&b=2` collapse to
one render. Prompted by the live DHW WebLink run, where breadcrumb links produced `Browse.aspx`/`browse.aspx`
case variants that wasted the page budget. Verified: dedup key equal across case/param-order under the flag,
distinct without it, and genuinely different `id=` folders stay distinct (value case preserved). Confirmed
on the live site that WebLink entries render as real `<a href="‚Ä¶Browse.aspx?id=N‚Ä¶">` anchors, so the
live-DOM `a[href]` harvest reaches them with no onclick tuning needed. (At the time, `crawl-render.js` read
CLI flags only; AD-084 then gave discover config parity with the GUI.)

## AD-084: Discover honors the GUI's `crawl-gui-config.txt` limits

## AD-086: Laserfiche document mode (`--laserfiche`) ‚Äî scan documents, don't render viewers
**Date:** 2026-06-28
**Problem:** On the live DHW WebLink site the pipeline rendered every page but never scanned a link *inside* a
document. Two causes: (1) document scanning is the verify pass (phase 2), which only starts after discover
finishes ‚Äî but more fundamentally (2) Laserfiche serves a document as a `DocView.aspx?id=N` **viewer page**,
not a file URL. `looksLikeDocument` (extension-based) classifies those as ordinary HTML pages, so discover
*recursed into and rendered every one* (thousands of wasted browser renders on a big repo), and when `crawl.js`
later fetched `DocView.aspx?id=N` it got the viewer's HTML, not the document ‚Äî nothing to scan. Per Laserfiche's
docs the file bytes come from the same URL with a download param (`?openpdf=true` for image docs / `?openfile=true`
for native files).
**Decision:** add `--laserfiche` (and `--laserfiche-dl`, default `openpdf=true`) to discover. A new
`isLaserficheDoc(u)` (path ends `/DocView.aspx` + an `id` param, case-insensitive) is checked in the link
classifier *before* the page/recurse branch: when on, the viewer URL is rewritten via `laserficheDownloadUrl`
(sets the download param) and recorded as a **document** ‚Äî so it lands in the seeds as a fetchable file and is
**not rendered**. The result: discover renders only folder (`Browse.aspx`) pages, and `crawl.js --seeds` fetches
the `?openpdf=true` URLs, whose `application/pdf` response trips `crawl.js`'s existing content-type/‚Äãmagic-byte
doc-scanner ‚Äî so links inside the documents are finally extracted and verified. When discover sees a DocView link
*without* the flag it prints a one-line hint suggesting `--laserfiche`. Exposed in the GUI as an on-by-default
**Laserfiche document mode** checkbox (`laserfiche`, sub-option of Discover (JS site)); harmless on non-WebLink
sites since no URL matches. Settable via `crawl-gui-config.txt`.
**Verification:** unit tests for `isLaserficheDoc`/`laserficheDownloadUrl` (DocView vs Browse; case/param-order;
no-duplicate-param; openpdf/openfile) pass. Full in-process E2E against a WebLink-like fixture (folder pages
linking to `DocView.aspx?id=N` + a subfolder): with `--laserfiche` discover rendered **only the 2 Browse folders**
(zero DocView viewer renders), recorded the **3 documents as `?openpdf=true` URLs**, and a real `crawl.js --seeds`
pass fetched them and extracted **all 3 in-document links**. HTA JScript compiles (890 lines). README + `--help`
+ config example updated.
**Addendum (2026-06-28) ‚Äî default changed to `electronicfile` after live confirmation:** on DHW, `openpdf=true`
was ignored (export-reason requirement) and `openfile=true` just loaded WebLink's embedded PDF.js *viewer page*
(HTML, not the file). The operator confirmed the working endpoint is `ElectronicFile.aspx?docid=N&dbid=D&repo=R`
(the documented native-file download ‚Äî and precisely the documents that *have* links inside them). So
`laserficheDownloadUrl` now defaults to a **path swap + param rename** (`DocView.aspx?id=N` ‚Üí `ElectronicFile.aspx?docid=N`,
keeping dbid/repo), with `--laserfiche-dl openpdf=true|openfile=true` still available as the query-param form for
other instances. E2E re-run with an `ElectronicFile.aspx?docid=N` fixture endpoint (Content-Disposition attachment,
`application/pdf`): folder pages rendered, viewers not, 3 docs ‚Üí 3 in-document links scanned.

## AD-088: Multi-host scope, verify page cap, and GUI stat phase reset
**Date:** 2026-06-28
**Problem:** Three interconnected bugs surfaced during live Laserfiche testing with the DHW repository:
1. **Scope anchored to seeds[0] only.** When the user's Start URLs field contained both
   `healthandwelfare.idaho.gov` (main site) and `publicdocuments.dhw.idaho.gov` (the WebLink repo),
   `seeds[0]` was the main site, which anchored the discover scope to it. The entire WebLink repo was
   classified as "external" and never crawled. Discover was crawling the main DHW site (getting blocked on
   disease-condition pages) while finding nothing in the document repository.
2. **Verify command missing `--max-pages` cap.** `buildVerifyCommand()` omitted the pages flag, so
   `crawl.js` silently defaulted to its 200-page limit. A repository with thousands of seed documents
   would stop at 200 with no warning.
3. **GUI live stats mixed discover + crawl phases.** The GUI accumulated blocked/error counts from both
   the discover phase (pages rendered) and crawl.js verify phase (links inside pages), inflating the
   visible blocked count and making it diverge from the crawl report. In the live run, 34 "blocked" in
   the GUI were all discover-phase blocks on the wrong site; the crawl report showed 0.
**Decision:**
1. In `runDiscover()`, replace the single `start.hostname` scope check with `seedHosts` ‚Äî a `Set` of all
   seed hostnames. `isInternalHost()` iterates the set (with optional subdomain expansion). A console note
   appears when seeds span multiple hosts. The scope/path anchor (`start = seeds[0]`) is unchanged; only
   the host-classification logic now considers all seeds.
2. `buildVerifyCommand()` now hard-codes `--max-pages none` in the initial flag array. The seed list is
   already finite (produced by discover); depth is 0 (no new crawling); there is never a good reason to
   cap it.
3. `crawl-render.js` emits `# discover start <ISO> seeds=N` to the log at the very start of discover.
   `processLine` in the HTA detects this to set `gDiscoverPhase = true`, then resets all live counters
   (`gCrawled`, `gGood`, `gBroken`, `gBlocked`, ext, docs) when it subsequently sees `# crawl start`
   (only if `gDiscoverPhase` is true, so normal single-phase crawls are unaffected). `resetLiveState()`
   now also clears `gDiscoverPhase`.
**Verification:** manual code review ‚Äî `isInternalHost` covers single-host (same as before), same-subdomain,
cross-host seeds, and subdomain expansion. `buildVerifyCommand()` preview string now includes `--max-pages none`.
Stat-reset path: confirm `gDiscoverPhase` starts false, sets on `# discover start`, resets all stats and clears
on `# crawl start`; regular crawls see `# crawl start` with `gDiscoverPhase === false` so do not reset.

## AD-089: Fix instant "Done ‚Äî 0 crawled in 0:00" and discover-phase activity label
**Date:** 2026-06-28
**Problem:** After the AD-088 fixes, a second live test showed "Done ‚Äî 0 crawled (0 good, 0 broken) in 0:00"
immediately on run-start, with the live log displaying old content from a previous crawl. Two related root causes:
1. **Stale `DONE_` marker in the live log.** `startCrawl` cleared the live log with `deleteIfExists(livePath)`,
   which silently swallows file-deletion errors (`catch(e){}`). When deletion silently failed the old
   `DONE_0` from the previous run remained at the end of the file. With `gPos` reset to 0, the first poll
   (500 ms after launch) read the entire old file, found `DONE_0`, set `gDoneCode = 0`, stopped the timer,
   and displayed "Done ‚Äî 0 crawled in 0:00" before discover had written a single line.
2. **Old errPath content shown as liveout fallback.** When `livePath` IS empty (correctly deleted), the poll
   shows `errPath` content if livePath has no data yet. Old `errPath` from a previous run then appeared
   instead of fresh discover output.
3. **AD-088 counter-reset reverted.** The `# crawl start` handler that reset all live counters was reverted
   because it transiently showed "0 crawled" during the gap between the reset and crawl.js processing its
   first page, which users read as the crawl stopping. The reset was not reinstated.
**Decision:**
1. Replace `deleteIfExists(livePath)` with `writeFile(livePath, "")` in all three run-launch paths
   (`startCrawl`, `recheckLinks`, `rebuildReport`). FSO `OpenTextFile(path, 2, true)` (ForWriting) truncates
   an existing file to 0 bytes; it throws (caught by the outer try) when truly inaccessible, giving an error
   message rather than silently leaving a stale `DONE_` in place.
2. Add `deleteIfExists(errPath)` to each launch path before the bat runs, so the errPath fallback shows only
   fresh output from the new run.
3. Restore the `# crawl start` handler in `processLine` ‚Äî but now it only clears `gDiscoverPhase = false`
   and returns; it does not reset any counters. The accumulated discover count stays visible in the UI and
   the counter continues from where it left off.
4. Improve the activity label during the discover phase: show "discovering JS links‚Ä¶" instead of "crawling‚Ä¶"
   when `gDiscoverPhase` is true; when stalled at 0 pages, show "browser rendering, waiting for first page‚Ä¶"
   to signal that the delay is expected (browser open + JS settlement, not a hung crawl).
**Verification:** code trace ‚Äî `writeFile(livePath, "")` creates an empty file so the first poll reads 0 bytes
(returns ""); old `DONE_` can never be seen before the new run starts writing. `errPath` deletion removes stale
fallback content. `gDiscoverPhase` label flow: false ‚Üí true on `# discover start` ‚Üí false on `# crawl start`,
with the appropriate activity string rendered in each state.

**Problem:** After AD-089 fixed the *false* instant finish, the real symptom persisted: pointing the GUI at the

   "nothing crawled" becomes a self-explaining message.
**Verification:** bundled Chromium against three fixtures ‚Äî *jsanchors* (real `<a href>` injected by JS after
load): 5 documents before AND after (no regression); *iframe* (listing inside an `<iframe>`): **0 ‚Äî> 5
documents** after the fix; *clickrows* (URL computed in a JS handler, genuinely unharvestable): the diagnostic
fires. Full chain for the formerly-broken case ‚Äî discover(browser, iframe) ‚Äî> seeds ‚Äî> `crawl.js` ‚Äî> **6 pages,
0 errors**. `node --check` clean; no smart-quote corruption (applied ¬ß5 #31's tokenizer check after each Edit).

## AD-091: crawl.js treats all --seeds hosts as internal ‚Äî port AD-088 from discover into the verify engine
**Date:** 2026-06-28
**Problem:** The crawl report listed `publicdocuments.dhw.idaho.gov` (the WebLink repository ‚Äî itself a seed) under
**External destinations**, "found on" the main-site `healthandwelfare.idaho.gov` pages. crawl.js decides
